# Architecture Overview

FastTarot is an **npm-workspaces monorepo** with one shared core and two UI apps. The
guiding principle: **everything platform-agnostic lives in `@fasttarot/core`; each app
contains only its own rendering, gestures, and navigation.**

```
┌─────────────────────────────────────────────┐
│              @fasttarot/core (TS)              │
│  data · i18n resources · logic · tokens      │
└───────────────┬───────────────┬──────────────┘
                │               │
        ┌───────▼──────┐ ┌──────▼────────┐
        │   apps/web   │ │  apps/mobile  │
        │ React + Vite │ │ Expo + RN     │
        └──────────────┘ └───────────────┘
```

## Packages

### `packages/core`

Framework-agnostic. No React, no DOM, no React Native imports — so both apps can consume
it and it stays unit-testable in plain Node.

| Module | Responsibility |
|---|---|
| `data/tarot-cards.json` | Canonical 78-card dataset. Generated by `scripts/gen_cards.py`. |
| `src/types.ts` | `TarotCard`, `Language`, `Suit`, `Orientation`, `RevealedCard`, … |
| `src/tokens.ts` | Design tokens: `colors`, `fonts`, `MAX_SELECTION`/`MIN_SELECTION`. |
| `src/logic/deck.ts` | The immutable `DECK`, `getCardById`, `shuffle` (injectable RNG). |
| `src/logic/selection.ts` | `toggleSelection` (enforces the 1–10 rule), `canConfirm`, `isAtMax`. |
| `src/logic/reveal.ts` | `randomOrientation` (50/50), `revealCards`. |
| `src/logic/rotation.ts` | Seamless-ring wheel math: `makeWheelLayout`, `cardBaseAngle`, `cardAngle`, `cardTransform`, `dragToRotation`, `wrapRotation`, `isCardVisible`. |
| `src/i18n/` | i18next `resources` + `revealedCardLabel` / `cardName` formatters. |

Pure functions take an optional `rng: () => number` argument so tests are deterministic
(see `packages/core/test/core.test.ts`).

### `apps/web`

React + Vite + Tailwind + Framer Motion.

- `src/components/CardWheel.tsx` — binds pointer events to the shared rotation math; renders only the visible arc of card backs.
- `src/components/CardBack.tsx` — the celestial mandala as inline SVG.
- `src/components/RevealPage.tsx` — grid of revealed cards with a flip animation.
- `src/components/{LanguageSwitcher,Toast,EnterButton}.tsx`
- `src/App.tsx` — the pick → reveal screen state machine.
- `src/cardImages.ts` — resolves `imageKey` → bundled URL via `import.meta.glob`.
- `src/i18n.ts` — initializes i18next with `@fasttarot/core` resources; persists choice to `localStorage`.

### `apps/mobile`

Expo + expo-router + Reanimated + Gesture Handler + react-native-svg.

- `app/_layout.tsx` — gesture root, font loading, headerless stack.
- `app/index.tsx` — picker screen. `app/reveal.tsx` — reveal screen.
- `src/components/*` — native mirrors of the web components (same names, same behavior).
- `src/cardImages.ts` — **static** `require()` map (RN can't do dynamic image paths); generated by `scripts/gen_image_map.py`.
- `src/revealStore.ts` — tiny in-memory hand-off of selected ids between routes.
- `src/i18n.ts` — initializes i18next, defaulting to the device locale via `expo-localization`.

## Shared vs. duplicated

**Shared (write once in core):** the 78 cards, all translations, selection rules, reveal
orientation, and the wheel geometry. A change to the 10-card limit or the wheel spacing is
a one-line edit in core that both apps pick up.

**Duplicated per platform (unavoidable):** the rendering layer. Web uses DOM + Framer
Motion + `import.meta.glob`; mobile uses RN views + Reanimated + static `require()`. The
components are intentionally kept name-for-name parallel so they're easy to diff.

## Data flow (both apps)

```
DECK (core) ──▶ CardWheel renders backs on the visible arc
                    │  tap
                    ▼
             toggleSelection (core) ──▶ selected: string[]
                    │  (11th tap → rejected → Toast)
                    ▼
             canConfirm → EnterButton fades in
                    │  press
                    ▼
             revealCards (core) assigns 50/50 orientation, once
                    ▼
             Reveal screen: image (rotated 180° if reversed)
                          + revealedCardLabel (core, localized)
```

## Why this split

- **Correctness lives in one place.** The tricky bits — the 1–10 rule, the reveal
  randomization, the rotation trig — are pure functions with unit tests, not tangled into
  view code twice.
- **Consistency for free.** Both apps show identical card names, identical messages, and an
  identical card back, because they import the same modules.
- **Testable.** Core has no platform deps, so `node --test` runs the logic without a browser
  or simulator.
